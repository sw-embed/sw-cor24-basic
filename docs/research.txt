Thought

Thought for 16s
You want a 1977-inspired BASIC interpreter that:

runs on your p-code VM, not directly on COR24,

is small, interactive, and educational,

exposes PEEK/POKE for low-level access,

fits a 24-bit integer-only machine,

can coexist cleanly with your Pascal→p-code toolchain,

and ideally helps validate the VM as a second front end, not just as an end-user language.

That is a very good fit for the layered design you were already converging on: generic VM/runtime first, languages above it. Your earlier notes already point toward keeping the VM language-neutral, with generic allocation, memory movement, traps, debug hooks, and primitive I/O below Pascal-specific behavior; BASIC slots naturally into that same model as another front end/runtime layer rather than as a special case.

High-level design choice
There are two plausible designs.

Option A — classic interpreted BASIC
Store tokenized program lines and interpret them line-by-line.

Pros

Pros

PEEK / POKE design
This is one of the most important parts.

Semantics
PEEK(addr)
Returns a value from memory.

POKE addr, value
Writes a value to memory.

Width problem
Because your machine is 24-bit, decide what “one memory unit” means.

You have three plausible meanings:

Option A — byte-based
PEEK(addr) reads 8 bits

POKE addr,val writes 8 bits

Option B — word-based
reads/writes 24-bit words

Option C — both, with extensions
PEEK/POKE are byte-based

DEEK/DOKE or PEEKW/POKEW for words

Recommendation
Use byte-based PEEK/POKE, then add word access separately.

Why
historically familiar

best for UART/MMIO

best for character and raw hardware work

avoids ambiguity on non-word-aligned addresses

Then add one of:

PEEKW(addr)

POKEW addr,val

or

PEEK24(addr)

POKE24 addr,val

for your native word size.

Safety policy
This matters a lot.

In debug mode
bounds-check RAM access

trap writes to forbidden regions unless explicitly enabled

optionally distinguish code/data/MMIO

In raw mode
allow full access

Best compromise
Add a mode flag:

SAFE

UNSAFE

or a runtime command:

SYSTEM
MONITOR
But for v1, just document that PEEK/POKE can hit MMIO and code/data and are intentionally dangerous.

BASIC will likely be an interpreter implemented on top of the VM,

and if interpreters need repeated low-level services, those services should live in the VM as general primitives, not as ad hoc BASIC-only hacks.

That is the right direction.

Recommendation
Treat the VM as a portable abstract machine for both compiled and interpreted languages.

That means:

do not add BASIC-specific opcodes,

do add general interpreter-assisting primitives when they are broadly useful,

and keep language policy in the language runtime/interpreter layer.

So the rule should be:

Add a VM feature only if it is:
useful to more than one language,

fundamental enough that it belongs in the abstract machine,

significantly simpler/faster/safer in the VM than in p-code library code,

and stable enough that you are unlikely to regret the abstraction.

That gives you “do the right thing now” without polluting the VM.

The architectural split
I would formalize the split like this.

Layer 0 — host/target substrate
COR24 emulator, hardware, MMIO, terminal, tape/file abstractions.

Layer 1 — p-code VM
Language-neutral abstract machine.

Owns:

control flow

call/return

stack/frame model

globals

heap/raw memory

byte/word access

traps/exceptions

system call / device gateway

optional generic string/block helpers

optional generic interpreter helpers

Layer 2 — language runtimes
Pascal runtime, BASIC runtime, future Lisp runtime, etc.

Owns:

language-visible builtins

data representations beyond VM primitives

line storage

symbol tables

interpreter state

parser/tokenizer

Layer 3 — language implementations
Pascal compiler, BASIC interpreter, future interpreters/compilers.

That keeps the VM clean while still letting it grow in the right direction.

The key design question
The real question is not:

“Should BASIC get special support?”

It is:

“What VM primitives make interpreted languages first-class citizens on this abstract machine?”

That is a much better question.

What kinds of VM support are worth adding?
There are a few categories.

1. Memory and data movement primitives
These are almost always worth having if not already present.

Examples:

byte load/store

word load/store

indirect load/store

block copy

block fill/zero

compare blocks

bounds-checked memory access or trap-capable memory access

Why they matter
Interpreters do a lot of:

token scanning

buffer movement

line insertion/deletion

string copying

table walking

You can write all of that in p-code, but a few good primitives make every interpreter and runtime better.

Strong recommendation
If missing, add:

LOADB, STOREB

MEMCPY

MEMSET

maybe MEMCMP

These are not BASIC-specific at all.

2. Trap and error primitives
Interpreters need structured failure.

Examples:

TRAP code

TRAP_IF_ZERO

TRAP_IF_NEG

ASSERT-style conditional trap

optional error/status register or last-trap code

Why they matter
Useful for:

syntax/runtime error handling

bounds checks

illegal token detection

safe PEEK/POKE

debugger integration

Strong recommendation
If you do not already have a clean trap model, add one now.

This benefits Pascal too.

3. Generic system/device call gateway
Interpreters often need device access, terminal I/O, tape/file I/O, time, etc.

Examples:

SYSCALL id

HOSTCALL id

IOCALL id

with arguments passed on stack or in a small ABI-defined block.

Why they matter
This is much better than adding opcodes like:

PRINT_CHAR

READ_LINE

TAPE_READ

because those are too policy-specific.

Strong recommendation
Have one generic VM gateway for host/OS/device services.

Then BASIC, Pascal runtime, and future interpreters all use it.

4. Generic string/block scanning primitives
This is where you need judgment.

Interpreters often spend time doing:

skip spaces

scan digits

scan identifiers

compare keywords

find line end

parse delimiters

You could add VM support here, but this is where VM bloat starts if you are not careful.

My recommendation
Do not add “parse keyword” or “scan identifier” opcodes.

Those are too language-shaped.

But a few low-level helpers may be justified if heavily reused:

find byte in block

compare block to block

length until terminator

uppercase/lowercase conversion helper if your environment strongly wants it

Borderline examples
Possibly useful:

FIND_BYTE

STRLENZ

CMPSZ

But only if they are generic block/string ops, not “BASIC token” ops.

5. Frame/continuation/interpreter dispatch helpers
This is the most interesting area.

An interpreter often does repeated dispatch like:

fetch token

jump to handler

return to dispatch loop

If your VM is stack/call oriented, normal call/return may be enough.

But there are a few generic features that can help a lot.

Possible features
computed jump / indirect branch

jump table support

cheap dispatch primitive

coroutine/continuation-friendly call mechanism later

Recommendation
If the VM does not already have a decent indirect branch/jump-table capability, consider it.

Why:

useful for interpreters,

useful for virtual dispatch,

useful for future bytecode machines hosted on your p-code VM,

not BASIC-specific.

A generic opcode like:

JMP_IND
or

CALL_IND

can be very valuable.

6. Stream/iterator primitives
Since you want teletype/paper tape style systems, interpreters will often read from sequential streams.

This suggests one of two approaches.

Option A — implement streams entirely in runtime code
The runtime uses generic system calls to read/write bytes.

Option B — add VM-level generic stream primitives
Examples:

open stream

read byte

write byte

status

close

Recommendation
I would only add these at the VM level if you believe multiple languages will want stream semantics, which is likely.

But even then, I would still probably expose them via generic system calls, not dedicated opcodes.

So:

VM has a generic callout mechanism,

runtime libraries define “reader”, “punch”, “console” abstractions on top.

That keeps the VM simpler.

What should not go into the VM
These should stay out.

Do not add:
BASIC_NEXT_LINE

BASIC_LOOKUP_LINE

BASIC_PARSE_NUMBER

BASIC_PRINT

BASIC_INPUT

BASIC_FOR_SETUP

BASIC_GOSUB

keyword tokenization opcodes

line-number search opcodes

These are interpreter/runtime policy, not abstract-machine primitives.

Even if they are tempting now, they will make later interpreters awkward.

A good rule of thumb
Ask of each proposed opcode:

“Could Pascal runtime, BASIC, Lisp, and a future command shell all plausibly use this?”
If yes, it is a good VM candidate.

If no, it probably belongs in runtime/interpreter code.

My recommended VM extensions
Given your stated goals, this is the set I would evaluate first.

Definitely worth having if missing
Memory primitives
LOADB

STOREB

LOADW / native word load if not already clear

STOREW

MEMCPY

MEMSET

Control / dispatch
JMP_IND

maybe CALL_IND

Trap / safety
TRAP code

conditional trap support

clear machine-visible trap status

Generic host/device interface
one SYSCALL / HOSTCALL mechanism

These are broadly useful and future-proof.

Possible second-tier extensions
Only if usage proves it.

Block/string helpers
MEMCMP

FIND_BYTE

STRLENZ

Arithmetic helpers
If interpreters need them often:

divide with quotient/remainder pair

compare helpers

bitfield operations

Allocation helpers
generic block alloc/free if the VM does not already provide a solid heap API

What this means for BASIC
For BASIC itself, I would implement the interpreter mostly in ordinary p-code, using:

byte/word memory ops

generic call/return

generic host/device calls

generic traps

maybe indirect dispatch

Then BASIC’s own code handles:

line storage

tokenization

expression parsing

line-number lookup

FOR/NEXT

GOSUB/RETURN

PEEK/POKE

That is the right level.

Benefits of this approach
Pros
1. Cleaner VM
The VM becomes better at hosting many languages, not just BASIC.

2. Better long-term reuse
Future interpreters benefit:

BASIC

Lisp

shell/scripting language

monitor command language

macro processor

debugger expression evaluator

3. Less refactoring later
You avoid baking parser logic into the VM and then regretting it.

4. Better debugger story
Generic traps, memory ops, and indirect dispatch are easy to expose in the debugger.

5. Better performance where it matters
You accelerate the low-level hot spots without hardcoding a single language.

Cons
1. Slightly more VM design work now
You need to think carefully about ABI and semantics.

2. Risk of premature abstraction
If you add too many “generic” helpers, you may still end up with clutter.

3. Some interpreter code will still be verbose
That is fine. Not everything belongs in the VM.

Practical selection methodology
Before changing the VM, I would do this.

Step 1 — implement a rough BASIC interpreter design using current ops
Do not optimize yet. Just sketch the runtime.

Step 2 — identify repeated pain points
Count where the interpreter would repeatedly need:

byte scanning

table dispatch

memory movement

host I/O

trap handling

Step 3 — extract only the repeated language-neutral pain
Those become VM extension candidates.

Step 4 — test those same candidates against other future languages
Ask:

would Lisp want this?

would a monitor shell want this?

would a debugger expression evaluator want this?

would Pascal runtime want this?

If yes, promote to VM.

This gives you discipline without deferring forever.

My concrete recommendation to “do the right thing now”
If I were making the call today, I would not extend the VM with anything BASIC-shaped.

I would make sure the VM has this minimum interpreter-friendly base:

Core interpreter-friendly VM feature set
byte-addressable memory access

block copy/fill

generic trap/error primitive

generic host/device call primitive

indirect jump/call support

clean heap/block allocation API if dynamic buffers are needed

That is the sweet spot.

Everything above that should stay in interpreter/runtime code until multiple language implementations prove a more specific abstraction is truly general.

One sentence summary
You are right to want VM support for interpreter-assisting operations, but the right boundary is: add only language-neutral primitives to the VM, and keep language semantics, parsing, and control structures in the interpreter/runtime layer.
